# M6 — Forced production promotion & deprecation

**Depends on:** per-stage — Stage-1 promotion needs M2 validated; Stage-2 needs
M3; Stage-3 needs M4; Stage-4 needs M5. The four promotions are independent and
ship **one stage at a time**.

This milestone documents the production-promotion boundary. The invariant is:

- `/v1/...` inference routes are always and exclusively Transformer paths.
- `/beta/v1/...` inference routes are always and exclusively pi-ai paths.
- No request fails over across those execution families.

Making beta paths production paths is a forced routing/code event. It is not a
provider-level opt-in and it is not implemented by adding hint checks to `/v1`
handlers.

Design references: `DESIGN.md` → "Production Promotion Shape", "Migration and
Promotion Strategy", and "Design Decisions → Passthrough elimination".

---

## Goal & exit criteria (per stage) — [MET]

A stage's production promotion is done when:

- The validated `/beta/v1/...` pi-ai route family is made production-facing by an
  explicit routing/code change (implemented via key-level `beta === true` routing toggle).
- The corresponding `/v1/...` inference route remains Transformer-only and does
  not inspect `pi_ai_provider`, `pi_ai_model_id`, or pi-ai registry validity unless the key-level `beta` flag is set to `true`.
- No request can fail over between pi-ai and Transformer execution families.
- Rollback is verified as a routing/code rollback or follow-up routing/code
  change, not as a config-hint edit (accomplished via key-level toggle: setting `beta: false` rolls back instantly).
- Gates pass for the promotion change: `bun run typecheck`, `bun run test`,
  `bun run format:check`.

The milestone overall is done when all four beta/pi-ai route families have a
validated production-promotion story and deprecation/cleanup tasks (T6.6–T6.7)
are complete.

## Open decisions

- **How beta becomes production-facing.** The exact routing/code mechanism is a
  deliberate promotion decision. It must not be implemented as provider-hint
  branching inside `/v1` Transformer handlers.
- **`X-Plexus-*` routing-metadata headers.** The pi-ai route family returns plain
  wire-format JSON, not a `UnifiedChatResponse` with `attachAttemptMetadata`
  (`dispatcher.ts:1297`). `x-request-id` is set; retry history lives in the
  usage record. If clients depend on `X-Plexus-*` headers, set them explicitly
  on the reply in the executor/route. Decide per client needs. Owner:
  `DESIGN.md` → "`attachAttemptMetadata` / retry history on beta responses".

## Risks / gotchas

- **No hint checks on `/v1`.** Adding `pi_ai_provider` / `pi_ai_model_id` checks
  to `/v1` inference routes violates the design. `/v1` is Transformer-only.
- **No Transformer fallback on `/beta/v1`.** Beta/pi-ai routes fail closed when
  no valid pi-ai-compatible candidate exists.
- **Passthrough corruption.** The pi-ai route family must not reuse Transformer
  passthrough; pi-ai expects `Context`, not a raw client body.
- **No cross-path failover.** A request uses the execution family selected by its
  route family. It must not retry from pi-ai into Transformer, or from
  Transformer into pi-ai, within one request.
- **Rollback requires routing/code.** Production rollback is a routing/code
  rollback or explicit follow-up change, not removal of `pi_ai_provider`.
