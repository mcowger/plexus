# M1 — Foundation: config, DB, repo, frontend, shared utils

**Depends on:** nothing. **Blocks:** M2–M6.

This milestone makes the two pi-ai hint fields exist end-to-end and extracts the
shared utilities every later stage needs. **No request path changes happen
here** — after M1, a provider can be *configured* with pi-ai hints, but nothing
yet routes through pi-ai for key-auth providers. That arrives in M2.

Design references: `DESIGN.md` → "Configuration Additions", "Shared
Infrastructure to Extract", and "Known Issues and Gaps" (registry coverage,
`streamSimple` option loss, base-URL stripping).

---

## Goal & exit criteria

Done when **all** of the following hold:

- `pi_ai_provider` (provider-level) and `pi_ai_model_id` (provider-model-level)
  are accepted by the config schema, persisted/round-tripped through the DB,
  and editable in the frontend.
- A Drizzle migration adds the two columns to both postgres and sqlite schemas.
- `beta/pi-ai-utils.ts` exists and exports `resolveBaseUrl`,
  `buildReasoningOptions`, `buildPiAiModel`, and re-exports `buildThinkingOptions`.
- `jsonSchemaToTypeBox` is exported from `transformers/oauth/type-mappers.ts`
  (or relocated to a shared module) so beta parsers can import it.
- Startup validation warns (non-fatally) when a configured
  `(pi_ai_provider, pi_ai_model_id)` pair is not in the pi-ai registry.
- Gates pass: `bun run typecheck`, `bun run test`, `bun run format:check`,
  `bun run lint:migrations`.

### T1.5 — Create `beta/pi-ai-utils.ts` (shared utilities)

**New file:** `beta/pi-ai-utils.ts`. Source the behaviour from `run.ts` on the
PoC branch and from `oauth-transformer.ts` on `main`.

Export three functions (signatures per `DESIGN.md` → Shared Infrastructure):

- **`resolveBaseUrl(apiBaseUrl, upstreamApi, incomingApiType)`** — resolves the
  `string | Record<string, string>` union (when a record: key first by the
  upstream pi-ai API, then by known Plexus aliases for that upstream API, then
  `"default"`, then first value), then applies the SDK base-URL rule:
  strip a trailing bare `/v\d+` **only** for `anthropic-messages` (the Anthropic
  SDK appends `/v1` itself); preserve the full URL for all other API types.
  This logic is currently inlined in the PoC `run.ts` — lift it here. Do not use
  the client route's `incomingApiType` as the primary key: a chat-completions
  request can route to an Anthropic, Responses, or Gemini upstream via pi-ai.
- **`buildReasoningOptions(piApi, piModelId, effort?)`** — when `effort` is
  present, delegate to `buildThinkingOptions`. When absent, explicitly disable
  thinking per provider: `{ thinkingEnabled: false }` for `anthropic-messages`,
  `{ thinking: { enabled: false } }` for `google-generative-ai`, `{}` for the
  OpenAI family. (Mirrors what `streamSimple` does internally; prevents silent
  thinking-token spend.)
- **`buildPiAiModel(providerConfig, piAiProvider, piAiModelId, apiType)`** —
  call pi-ai `getModel(piAiProvider, piAiModelId)`, then return a shallow copy
  with `baseUrl` overridden to `resolveBaseUrl(providerConfig.api_base_url,
  piModel.api, apiType)`. This is the canonical "give me a ready-to-call pi-ai Model".

**Verify:** unit-test each function: `resolveBaseUrl` for the anthropic strip
vs. preserve cases and the record-union resolution; `buildReasoningOptions` for
the three providers with and without effort.

### T1.8 — Startup registry validation (non-fatal)

**Files:** `config.ts` (`hydrateConfig`, ~L1004) or a startup check consumed by
the executor.

- For each provider with `pi_ai_provider` set, and each of its models with
  `pi_ai_model_id` set, attempt `getModel(pi_ai_provider, pi_ai_model_id)`.
  `getModel` **throws** for unknown registry pairs (`DESIGN.md` → registry
  coverage).
- On failure, **log a warning, do not throw** — the pi-ai registry can lag new
  model releases. Runtime handling is path-specific: beta routes treat the pair
  as invalid for beta candidate filtering, while `/v1/...` inference routes do
  not inspect these hints. Catch the throw per pair.

**Verify:** unit test — a config with a bogus `pi_ai_model_id` hydrates without
error and emits a warning (assert via the logger mock).

## Risks / gotchas

- **Migrations are CI-gated.** Use `--name add_pi_ai_hint`; never edit the
  generated file; never run `drizzle-kit` directly.
- **Two dialects.** The postgres and sqlite schemas must both get the columns,
  or the migration generator / runtime will diverge.
- **`pi_model` vs `pi_ai_*` confusion.** They are different fields at different
  levels. Touching `pi_model` is out of scope.
- **`getModel` throws on miss.** The startup check must wrap each call in
  try/catch and downgrade to a warning (T1.8) — a thrown error here would
  prevent Plexus from starting.
- **Global pi-ai mock in tests.** `@earendil-works/pi-ai` is mocked in
  `test/vitest.setup.ts`; the mock's `getModel`/`getModels` shape must cover any
  fields `buildPiAiModel`/validation read (`baseUrl`, `api`). Extend the mock if
  a field is missing rather than re-mocking in the test file.
