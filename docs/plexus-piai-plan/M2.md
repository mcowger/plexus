# M2 — Stage 1: `/beta/v1/chat/completions` + the full pi-ai executor

**Depends on:** M1. **Blocks:** M3, M4, M5 (they reuse the executor and the beta
route pattern). M6 Stage-1 production promotion depends on this.

This is the keystone milestone. It builds the **stage-agnostic executor**
(`beta/pi-ai-executor.ts`) with the *complete* Dispatcher integration the PoC
left out, plus the OpenAI inbound/outbound and the first beta route. M3–M5 then
become "write a parser + serialiser + route" because the hard part lives here.

Design references: `DESIGN.md` → "The Beta Parallel Path Pattern", "Stage 1",
"Full Dispatcher Integration Design", "Raw response capture", "Known Issues and
Gaps". PoC reference (read, don't build on):
`feat/piai-parallel-chat-completions-inference-path:src/beta/{run.ts,openai-to-context.ts,context-to-openai.ts,index.ts}`.

---

## Goal & exit criteria

Done when:

- `beta/pi-ai-executor.ts` implements the full failover loop with cooldown
  pre-check, concurrency acquire/release (including release-after-stream),
  per-attempt timeout kept active through streaming completion, stall detection
  (TTFB for streaming via a real timeout race), retry classification, usage
  recording, explicit quota recording, and debug logging.
- `beta/openai-to-context.ts` and `beta/context-to-openai.ts` exist and pass
  unit tests for all message/event shapes.
- `POST /beta/v1/chat/completions` is registered (via `beta/index.ts` →
  `registerBetaRoutes`, wired into `routes/inference/index.ts` inside the
  protected block) and fails closed with HTTP 400 when no registry-valid
  beta-compatible candidate remains.
- Raw upstream response capture for debug works for both streaming and
  non-streaming (global `fetch` tap, recommended approach).
- Integration test: mock pi-ai `stream`/`complete` with known event sequences;
  assert SSE output and `UsageRecord` shape.
- Gates pass: `bun run typecheck`, `bun run test`, `bun run format:check`.

## Tasks

### T2.1 — Inbound parser `beta/openai-to-context.ts`

**New file.** Port and harden from the PoC. Converts OpenAI chat-completions
JSON → pi-ai `Context` + a `ProviderStreamOptions` fragment.

Must handle (`DESIGN.md` → Stage 1 inbound):
- `system` and `developer` roles → collapsed into `context.systemPrompt`
  (multiple system messages concatenated with `\n\n`; document this).
- `user` messages: text and base64 image content (`ImageContent`). **URL image
  content: keep rejecting with an error for now** (open decision below).
- `assistant` messages: text + tool calls (`ToolCallContent`).
- `tool` role messages → `ToolResultMessage`.
- Tool definitions via the now-exported `jsonSchemaToTypeBox` (from M1.7),
  imported from `transformers/oauth/type-mappers`.
- Options fragment: `temperature`, `max_tokens` / `max_completion_tokens`
  (→ `maxTokens`), `tool_choice`, `parallel_tool_calls`, and `reasoning_effort`
  → routed through `buildReasoningOptions` (M1.5) at execute time. **Use the
  `stream()`/`complete()` `ProviderStreamOptions` path, never
  `streamSimple()`** — the latter drops `toolChoice`/`parallelToolCalls`
  (`DESIGN.md` → "Why `stream()` and not `streamSimple()`").

**Verify:** unit tests covering every role combination, multi-part content,
tool defs, and reasoning_effort pass-through (`DESIGN.md` → Stage 1 testing).

### T2.2 — Outbound serialiser `beta/context-to-openai.ts`

**New file.** Port from the PoC. Converts `AssistantMessage` →
`OpenAIChatCompletion` and `AssistantMessageEvent` → `OpenAIChatChunk`, plus
`chunkToSSE` (emits `data: {...}\n\n`) and a terminal `data: [DONE]\n\n`.

Must handle (`DESIGN.md` → Stage 1 outbound):
- Text, thinking/reasoning (`reasoning_content`).
- Tool calls with the correct **0-based `tool_calls` array index**, kept
  separate from the content-block index (this was a real PoC subtlety).
- Usage field mapping.
- Stop-reason mapping.
- Error surfacing: when `AssistantMessage.stopReason === 'error'`, surface
  `message.errorMessage` as content so the caller sees the upstream error.

**Verify:** unit tests for all event types, tool-call index computation, usage
mapping, stop-reason mapping, and the error path.

### T2.3 — The executor `beta/pi-ai-executor.ts`

**New file.** This replaces the PoC's `run.ts` orchestrator with the
production design. Implement the interface and failover loop from `DESIGN.md` →
"Full Dispatcher Integration Design" (read that section in full; it contains
the loop pseudocode).

Interface (per design):
```
interface PiAiExecutorInput {
  requestId, incomingApiType, modelAlias, context, streamOptions,
  streaming, request, usageStorage, quotaEnforcer?, signal?, onSuccess
}
interface PiAiExecutorResult { response?: unknown; stream?: AsyncIterable<string> }
```

The executor is **wire-format-agnostic** — it never imports OpenAI/Anthropic
types. The route handler passes the parser output (`context`, `streamOptions`,
`streaming`) and the serialiser as callbacks. (Concretely: the executor takes
serialiser callbacks for non-streaming and streaming, or returns the raw
`AssistantMessage`/event stream and the route applies the serialiser — pick one
shape and keep it consistent for M3–M5. Recommended: executor takes
`serializeMessage` and `serializeEvents` callbacks so streaming TTFB/usage
tapping stays inside the executor.)

Sub-tasks (each maps to a verified symbol — file:line from baseline):

- **T2.3a — Candidate resolution & key policy.**
  `candidates = await Router.resolveCandidates(modelAlias, incomingApiType)`
  (`services/router.ts:267`); fall back to `[await Router.resolve(...)]`
  (`router.ts:398`) if empty; then `applyKeyAccessPolicy(...)`
  (`dispatcher.ts:1452` — currently private to Dispatcher; **open decision**
  below on how to share it). After key policy, filter to **beta-compatible**
  candidates only: both hints must be present and `getModel(pi_ai_provider,
  pi_ai_model_id)` must succeed. If the filtered list is empty, return the
  route's protocol-shaped 400/failed response. Do not include non-hint or
  registry-invalid candidates in the beta failover loop, and do not fall back to
  the legacy Transformer path.
- **T2.3b — Per-candidate guards.** For each candidate, in order:
  abort check → `createAttemptTimeout(signal, route.config.timeoutMs)`
  (`dispatcher.ts:2429`) → `CooldownManager.isProviderHealthy(provider, model)`
  (`cooldown-manager.ts:261`); on unhealthy, `appendSkippedAttempt`
  (`dispatcher.ts:1339`) and continue → `ConcurrencyTracker.acquire(provider,
  model)` (`concurrency-tracker.ts:33`); on false, skip + append + continue.
- **T2.3c — Build the call.** Read `route.config.pi_ai_provider` and
  `route.modelConfig?.pi_ai_model_id`; the candidate filter should already have
  removed missing or registry-invalid hints, but keep a defensive fail-closed
  check. Build the model with `buildPiAiModel(...)` (M1.5), which resolves
  configured base URLs by upstream pi-ai API (`piModel.api`), not by the client
  route type. Assemble `ProviderStreamOptions`:
  `{ ...streamOptions, ...buildReasoningOptions(...), apiKey: route.config.api_key,
  headers: route.config.headers, signal: attemptTimeout.signal, onPayload: (p)
  => debug.addTransformedRequest(requestId, p) }`.
- **T2.3d — Stall config.** Resolve global stall config merged with route
  overrides (`route.config.stallTtfbMs` etc.); the HTTP path uses
  `wireStallDetection` / `addStallConfig` (`utils/stall.ts:122`). For pi-ai
  there is **no `probeStreamingStart`** (it needs a raw `Response`); implement
  the TTFB equivalent in the stream generator (T2.4).
- **T2.3e — Invoke pi-ai.** Non-streaming: `await complete(piModel, context,
  options)`. Streaming: `stream(piModel, context, options)` returning an
  `AsyncIterable<AssistantMessageEvent>`, wrapped by the SSE generator (T2.4).
- **T2.3f — Success path.** Non-streaming: after `complete()` returns, call
  `CooldownManager.markProviderSuccess` (`cooldown-manager.ts:228`),
  `appendSuccessAttempt` (`dispatcher.ts:1356`), release slot,
  `attemptTimeout.cleanup()`, then `await onSuccess(message)`. Streaming:
  return the wrapped stream without marking success, releasing concurrency, or
  cleaning up timeout yet. The stream wrapper owns those actions after a
  successful terminal event; it must also own failure/cancel cleanup if the
  stream errors.
- **T2.3g — Failure path.** Compute `effectiveErr` (timeout vs. raw) via
  `attemptTimeout.isTimedOut()`; if `signal?.aborted` throw `buildCancelledError`
  (`dispatcher.ts:2461`) — client disconnect is **never** retried.
  Call `markProviderFailure` (`cooldown-manager.ts:175`) or, for stalls,
  `markProviderStallFailure` (`cooldown-manager.ts:93`) **only if**
  `route.config.stall_cooldown` is set (else skip cooldown for stalls, same as
  HTTP path). For failures before a stream is returned to the route, release the
  slot and clean up the attempt timeout here. For failures from an already
  returned streaming wrapper, the wrapper performs release/cleanup exactly once.
  Compute `canRetry = failoverEnabled && i < last && isRetryable(effectiveErr)`;
  `appendFailureAttempt` (`dispatcher.ts:1372`); if retry,
  `saveIntermediateError` (`dispatcher.ts:203`) and continue; else
  `throw buildAllTargetsFailedError(...)` (`dispatcher.ts:1395`).
- **T2.3h — `isRetryable(err)`.** Retryable = signal not aborted AND
  (`AbortError` from attempt timeout → `upstream_timeout`, OR network errors
  ECONNREFUSED/ETIMEDOUT, OR pi-ai HTTP errors surfacing as 5xx). Non-retryable
  otherwise. (`DESIGN.md` → Stage 1 "retryable errors".) For streaming, this
  predicate is additionally gated by whether any client frame has already been
  emitted: retry only before first emitted SSE/NDJSON frame. Use
  `extractPiAiErrorMessage` (`type-mappers.ts:560`) for messages.

**Verify:** unit-test the loop with a mocked Router/Cooldown/Concurrency:
healthy→success, unhealthy→skip→next, acquire-fail→skip, retryable→failover,
non-retryable→throw, all-exhausted→`buildAllTargetsFailedError`, abort→cancelled.

### T2.4 — Streaming generator + TTFB stall + concurrency release

Inside the executor (the function the design calls `buildSSEStream`):

- Wrap `for await (const event of eventStream)`; start a TTFB deadline when
  `stream()` returns. Implement this as a real `Promise.race` around
  `iterator.next()` (or an equivalent helper), not as a wall-clock check inside
  the loop body. If no non-`start` event arrives within `stallConfig.ttfbMs`,
  abort the upstream via the signal and throw a `StallError` that the failover
  loop treats as a stall only if no client frame has been emitted yet
  (`DESIGN.md` → "Stall detection for pi-ai streaming" and "Streaming failure
  boundary").
- Accumulate content blocks + usage as events arrive (for the reconstructed
  debug response, T2.6).
- **Streaming lifecycle in terminal path:** keep concurrency and the per-attempt
  timeout active until the stream terminal event, client cancel, or error. On a
  successful terminal event, mark provider success, append success history,
  save usage, record quota usage, call `onSuccess`, release the slot, clean up
  timeout, and `debug.flush(requestId)`. On error/cancel, release and clean up
  exactly once, save error usage/debug state, and do not retry if a client frame
  has already been emitted.
- Yield serialised frames (`chunkToSSE(...)` + terminal `SSE_DONE`).

**Verify:** unit test a stream that stalls before first content → aborts and is
eligible for failover; a stream that fails after the first client frame does not
retry; a normal stream releases the slot and cleans up timeout exactly once.

### T2.5 — Beta route `beta/index.ts` + wiring

**New file** `beta/index.ts` exporting `registerBetaRoutes(fastify,
usageStorage, quotaEnforcer?)`. Port the route scaffolding from the PoC
`index.ts` but call the new executor.

Handler for `POST /beta/v1/chat/completions`:
1. `requestId = crypto.randomUUID()`; set `x-request-id` header.
2. `debug.startLog(requestId, body, sanitizeHeaders(request.headers))`.
3. Quota: if `quotaEnforcer`, `if (!(await checkQuotaMiddleware(request, reply,
   quotaEnforcer))) return;` (`quota-middleware.ts:18`).
4. Wire abort: `wireUpstreamTimeout` (`utils/timeout.ts:27`) +
   `wireEarlyDisconnectDetection` (`utils/timeout.ts:61`).
5. Parse: `openaiRequestToContext(body)` → `{ context, streamOptions, streaming }`.
6. Call executor; write `result.response` as JSON or pump `result.stream` into a
   `ReadableStream` with SSE headers (PoC pattern).
7. `onSuccess` is a no-op for Stage 1.
8. Error shape: OpenAI `{ error: { message, type } }`.

**Wire into app:** in `routes/inference/index.ts`, inside the protected
`fastify.register(async (protectedRoutes) => {...})` block (after
`registerResponsesRoute`, ~L37), add `await registerBetaRoutes(protectedRoutes,
usageStorage, quotaEnforcer);` so beta inherits auth + bearer auth.

**Verify:** route returns 400 when no registry-valid beta-compatible candidate
remains; returns SSE for `stream:true` and JSON otherwise (integration test,
T2.8).

### T2.6 — Usage recording

The executor populates a `UsageRecord` incrementally (`DESIGN.md` → "Usage
recording"; `types/usage.ts` / `services/usage-storage.ts`) and records quota
usage after costs are finalized:
- `emitStartedAsync(record)` at request start (`usage-storage.ts:176`).
- `emitUpdatedAsync({ requestId, provider, selectedModelName, canonicalModelName })`
  after route resolution (`usage-storage.ts:180`).
- `saveRequest(record)` after stream completes / non-streaming response built
  (`usage-storage.ts:120`).
- Fields from pi-ai: `tokensInput/Output/Cached/CacheWrite` from
  `AssistantMessage.usage`; `costInput/Output/.../Total` from
  `calculateCost(piModel, usage)`; `costSource: 'pi-ai'`; `finishReason` from
  `stopReason`; `toolCallsCount` = count of `ToolCallContent` blocks; `ttftMs` =
  start→first non-`start` event; `durationMs` = total; `attemptCount` /
  `retryHistory` from the loop. `responseStatus` ∈
  `success|error|timeout|cancelled`.
- After `saveRequest(record)`, if `quotaEnforcer` and `(request as any).keyName`
  are present, call `recordQuotaUsage(keyName, record, quotaEnforcer)` from
  `services/quota/quota-middleware.ts`. `saveRequest()` alone does not consume
  quota.

**Verify:** integration test asserts the saved `UsageRecord` matches expected
token/cost/finish/ttft fields for a known mocked event sequence, and that quota
usage is recorded once with the finalized token/cost values.

### T2.7 — Debug logging + raw-response capture

- Calls: `debug.startLog` (route, T2.5), `debug.setProviderForRequest(requestId,
  provider)` (`debug-manager.ts:103`) right after route resolution,
  `debug.addTransformedRequest` via `onPayload`, `debug.addTransformedResponse`
  for non-streaming and (for streaming) the **reconstructed** `AssistantMessage`
  after `done`, `debug.flush` in the generator `finally`.
- **Raw upstream response capture:** pi-ai does not expose the raw body.
  Implement the **global `fetch` monkey-patch** (recommended in `DESIGN.md` →
  "Raw response capture"): at server startup, before registering beta routes,
  wrap `globalThis.fetch`. Correlate the in-flight request to `requestId` via
  `AsyncLocalStorage` set in the `onPayload` callback. On response: for
  non-streaming clone+buffer the body → `DebugManager.addRawResponse`; for
  streaming wrap the body in a `TransformStream` that copies chunks to the debug
  log as they pass through (**not** `tee()` — avoids upstream backpressure).
  Only tap when debug is active for the current request.

**Verify:** with debug enabled, a non-streaming call records both transformed
request (via `onPayload`) and raw response; a streaming call records the
reconstructed response after `done`.

## Open decisions

- **Sharing `applyKeyAccessPolicy`** (currently private on `Dispatcher`,
  `dispatcher.ts:1452`). Options: (a) extract to a shared helper both
  Dispatcher and executor import; (b) duplicate the small policy in the
  executor. Recommended: (a) extract — avoids drift. Owner: `DESIGN.md` →
  Full Dispatcher Integration / failover loop.
- **Executor return shape**: serialiser-callbacks-inside vs. raw-message-out.
  Recommended: serialiser callbacks inside the executor so TTFB timing and
  reconstructed-debug stay co-located. Lock this in M2; M3–M5 must follow it.
- **URL image content**: keep the error (Stage 1). Revisit globally when a
  backend known to support URL images is configured. Owner: `DESIGN.md` →
  Stage 1 inbound "Gaps to close".
- **Raw capture approach**: global `fetch` tap (recommended) vs. local proxy.
  Pick the tap; it's simplest and Plexus is the only `fetch` caller in-process.

## Risks / gotchas

- **`streamSimple` option loss** — must use `stream()`/`complete()` with
  `ProviderStreamOptions`, or `tool_choice`/`parallel_tool_calls` vanish
  silently (`DESIGN.md` → known issues). The PoC's `run.ts` header comment says
  it uses `streamSimple`, but the design mandates `stream()`; **follow the
  design, not the PoC comment.**
- **Base-URL stripping** — only `anthropic-messages` strips trailing `/v\d+`
  (handled by `resolveBaseUrl` from M1). A new API type could need different
  handling; validate against a live provider.
- **Concurrency leak on streaming** — release must be in both `finally` and
  `catch` of the generator, never before consumption ends.
- **Timeout leak on streaming** — per-attempt timeout cleanup must also wait for
  stream completion/error; cleaning it up immediately after `stream()` returns
  leaves hung streams unbounded.
- **Post-first-frame retry corruption** — retrying after any SSE/NDJSON frame has
  reached the client mixes providers in one stream. Retry streaming attempts only
  before the first emitted client frame.
- **Quota accounting** — `saveRequest()` does not consume quota. Call
  `recordQuotaUsage()` after costs are calculated.
- **Stall semantics** — respect `route.config.stall_cooldown`: when false, a
  stall failure must **not** trigger a cooldown.
- **Client disconnect never retried** — gate retry on `!signal.aborted`.
- **`getModel` panics on registry miss** — M1.8 validates at startup, but the
  beta executor must still filter registry-invalid candidates before execution.
  If no valid beta-compatible candidates remain, fail closed with the route's
  protocol-shaped error; do not fall through to the legacy Transformer path.
- **Raw-capture global state** — the `fetch` tap touches process-global state;
  scope it to debug-active requests via `AsyncLocalStorage` and ensure tests
  don't leak the patched `fetch` between cases.
