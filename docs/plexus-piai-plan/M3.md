# M3 — Stage 2: `/beta/v1/messages` (Anthropic-compatible)

**Depends on:** M2 (executor + beta pattern). **Blocks:** M6 Stage-2 production promotion.

Anthropic is structurally different from OpenAI chat-completions, but the
executor from M2 is wire-format-agnostic, so this milestone is almost entirely
*parser + serialiser + route*. The executor, failover, cooldown, concurrency,
stall, usage, debug, and raw-capture are all inherited unchanged — only
`incomingApiType: 'messages'` and the error-shape differ.

Design references: `DESIGN.md` → "Stage 2". Existing code to study (do not
import wholesale): `transformers/anthropic/request-parser.ts`
(`parseAnthropicRequest`), `transformers/anthropic/stream-formatter.ts`
(`formatAnthropicStream`), `getThinkLevel` (`transformers/utils.ts:43`).

---

## Goal & exit criteria

Done when:

- `beta/anthropic-to-context.ts` converts Anthropic messages JSON → pi-ai
  `Context` + `ProviderStreamOptions`, including thinking blocks in history.
- `beta/context-to-anthropic.ts` emits the Anthropic non-streaming response and
  the full Anthropic SSE event sequence from `AssistantMessage` /
  `AssistantMessageEvent`.
- `POST /beta/v1/messages` is registered in `beta/index.ts`, uses the M2
  executor with `incomingApiType: 'messages'`, and returns Anthropic-shaped
  errors.
- Unit + integration tests pass (SSE byte-shape vs. the Anthropic protocol).
- Gates: `bun run typecheck`, `bun run test`, `bun run format:check`.

## Tasks

### T3.1 — Inbound parser `beta/anthropic-to-context.ts`

**New file.** Mirrors `openai-to-context.ts` in structure, speaks Anthropic.
Cross-check every mapping against the existing `parseAnthropicRequest()` so
behaviour matches the legacy path (`DESIGN.md` → Stage 2 inbound).

- **System prompt:** accept `string` and `ContentBlock[]`. For the array form,
  concatenate text blocks with `\n\n`. **Cache-control on system blocks:** see
  open decision — for now, pass through if pi-ai propagates it, else strip.
- **User messages:** content arrays — `text` → `TextContent`; `image` with
  `source.type === "base64"` → `ImageContent`; `image` with `source.type ===
  "url"` → same decision as Stage 1 (reject for now).
- **Assistant messages (history):** preserve `thinking` blocks. pi-ai
  `AssistantMessage.content` supports `ThinkingContent` (`{ type: "thinking",
  thinking }`). Anthropic **requires** thinking blocks in history when extended
  thinking is active, so they must survive the round-trip.
- **Tool use/result:** `tool_use` (assistant) → `ToolCallContent`;
  `tool_result` (user) → `ToolResultMessage`.
- **Tool definitions:** Anthropic `{ name, description, input_schema }`; map
  `input_schema` through `jsonSchemaToTypeBox` (exported in M1.7).
- **Reasoning:** Anthropic uses `{ thinking: { type: "enabled", budget_tokens:
  N } }`. Map `budget_tokens` → an effort string via `getThinkLevel`
  (`transformers/utils.ts:43`), then feed `buildReasoningOptions` (M1.5).

**Verify:** unit tests — multi-block system prompts, thinking blocks in history,
tool use/result interleaving, thinking-config mapping.

### T3.2 — Outbound serialiser `beta/context-to-anthropic.ts`

**New file.** Study `formatAnthropicStream()` in
`transformers/anthropic/stream-formatter.ts` for edge cases (multiple tool
calls, interleaved thinking + text) before finalising.

- **Non-streaming:** `{ id, type: "message", role: "assistant", model, content:
  [...], stop_reason, stop_sequence: null, usage }`. Content order: `thinking`
  blocks first (if present), then `text`, then `tool_use`. Stop-reason map:
  `stop`→`end_turn`, `toolUse`→`tool_use`, `length`→`max_tokens`. Usage:
  `{ input_tokens, output_tokens, cache_read_input_tokens,
  cache_creation_input_tokens }`.
- **Streaming (Anthropic SSE):** consume `AssistantMessageEvent` and emit
  (`DESIGN.md` → Stage 2 streaming, full event list):
  - `start` → `message_start` (empty content, initial usage if available).
  - `text_delta` → lazy `content_block_start {type:"text"}` then
    `content_block_delta {type:"text_delta"}`.
  - `thinking_delta` → lazy `content_block_start {type:"thinking"}` then
    `content_block_delta {type:"thinking_delta"}`.
  - `toolcall_delta` → on first delta `content_block_start
    {type:"tool_use", id, name, input:{}}` then `content_block_delta
    {type:"input_json_delta", partial_json}`.
  - `done` → `content_block_stop` for any open block, then `message_delta
    {stop_reason, stop_sequence:null}` with final usage, then `message_stop`.
  - `error` → `message_delta` with `stop_reason:"error"` and close.

**Verify:** unit tests for the full SSE sequence (text-only; thinking+text;
single tool; multi-tool) and the non-streaming shape.

### T3.3 — Route `POST /beta/v1/messages`

**File:** `beta/index.ts` (extend `registerBetaRoutes`).

- Same scaffold as `/beta/v1/chat/completions`: requestId + `x-request-id`,
  `debug.startLog`, quota middleware, abort wiring, parse via
  `anthropicRequestToContext`, call the M2 executor with `incomingApiType:
  'messages'`, write JSON or Anthropic SSE.
- **Error shape must be Anthropic:** `{ type: "error", error: { type, message }
  }` — not the OpenAI shape. (Matches the existing `/v1/messages` handler.)
- `onSuccess` is a no-op for Stage 2.

**Verify:** 400 when no registry-valid beta-compatible candidate remains
(Anthropic error shape); SSE vs. JSON by `stream` flag.

### T3.4 — Tests

- Integration: mock pi-ai with a known event sequence and assert the SSE output

## Open decisions

- **Cache-control on system blocks** (Anthropic prompt caching). Determine
  whether pi-ai's Anthropic SDK integration propagates `cache_control` on system
  blocks: if yes, pass the array form through; if no, the `\n\n` concatenation
  is acceptable (pi-ai manages caching at a different level). Owner: `DESIGN.md`
  → "Multi-system-prompt concatenation". **Resolve before finalising T3.1.**
- **URL image content** — same as Stage 1: reject for now.
- **Thinking detection robustness** — `buildThinkingOptions` currently uses
  fragile model-ID `includes()` checks. `DESIGN.md` ("Thinking options for
  non-thinking models") wants this hardened to read `piModel.compat` /
  `piModel.thinkingLevelMap` **before Stage 2 ships to production**. Decide
  whether to harden in this milestone (recommended, since Anthropic is the
  thinking-heavy provider) or track as a follow-up. If hardened, do it in
  `beta/pi-ai-utils.ts` (`buildThinkingOptions`/`buildReasoningOptions`) so all
  stages benefit.

## Risks / gotchas

- **Thinking blocks in history are mandatory** for Anthropic extended thinking —
  dropping them on the inbound side breaks multi-turn thinking conversations.
- **Content ordering** in the non-streaming response (thinking → text →
  tool_use) is part of the Anthropic contract; assert it in tests.
- **SSE event ordering** — Anthropic clients are strict about the
  `message_start` → `content_block_*` → `message_delta` → `message_stop`
  sequence; the lazy `content_block_start` emission must fire before the first
  delta of each block.
- **Error shape divergence** — easy to accidentally reuse the OpenAI error
  helper; the messages route must emit `{ type:"error", error:{...} }`.
- **Executor reuse** — do not fork the executor. If something is missing for
  Anthropic, it belongs in the shared executor (M2), not a Stage-2 copy.
