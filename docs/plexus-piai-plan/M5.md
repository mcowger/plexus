# M5 — Stage 4: `/v1beta/models/:model:generateContent` (Gemini-compatible)

**Depends on:** M2 (executor + beta pattern). **Blocks:** M6 Stage-4 production promotion.

Gemini is the most structurally distinct surface: the model name is URL-encoded
with an action suffix, streaming is detected from the URL (not the body),
streaming uses **NDJSON** (one JSON object per line) instead of SSE, and tool
results are encoded as `functionResponse` parts inside a `user`-role turn (the
trickiest mapping in the whole project). The executor is still reused unchanged;
the only executor-facing differences are `incomingApiType: 'gemini'`, the
`streaming` flag derived from the URL, and NDJSON framing on the serialiser
side.

Design references: `DESIGN.md` → "Stage 4". Existing code to study:
`routes/inference/gemini.ts` (route + action parsing: `:modelWithAction` at L28,
`streamGenerateContent` detection at L97), `transformers/gemini/`
(`request-parser.ts`, `part-mapper.ts`, `response-formatter.ts`,
`stream-formatter.ts`, `stream-transformer.ts`).

## Goal & exit criteria

Done when:

- `beta/gemini-to-context.ts` converts Gemini `contents` → pi-ai `Context` +
  `ProviderStreamOptions`, including the `functionResponse` → `ToolResultMessage`
  mapping.
- `beta/context-to-gemini.ts` emits the Gemini non-streaming object and **NDJSON
  streaming** frames.
- A beta Gemini route is registered (URL-encoded model + action), derives
  `streaming` from the `:streamGenerateContent` suffix, calls the M2 executor
  with `incomingApiType: 'gemini'`, and emits NDJSON for streaming.
- Unit + integration tests pass.
- Gates: `bun run typecheck`, `bun run test`, `bun run format:check`.

## Tasks

### T5.1 — Inbound parser `beta/gemini-to-context.ts`

**New file.** Cross-check mappings against `transformers/gemini/request-parser.ts`
and `part-mapper.ts` (`DESIGN.md` → Stage 4 inbound).

- **System instruction:** `systemInstruction.parts[].text` →
  `context.systemPrompt`.
- **Contents:** each `{ role: "user" | "model", parts: [...] }`. `role:"model"`
  → assistant. Parts: `text` → `TextContent`; `inlineData` (base64) →
  `ImageContent`; `functionCall` → `ToolCallContent`; `functionResponse` →
  `ToolResultMessage`. **The tricky case:** Gemini encodes tool results as
  `functionResponse` parts *inside a `user`-role turn* — detect these and split
  them out into `ToolResultMessage`s rather than treating the turn as a plain
  user message.
- **Tools:** Gemini `functionDeclarations` → pi-ai `Tool` via
  `jsonSchemaToTypeBox` (M1.7). Gemini uses OpenAPI-style schemas (mostly
  compatible; watch for type-name differences).
- **Reasoning:** Gemini 2.5+ uses `generationConfig.thinkingConfig`. Map to
  `buildReasoningOptions('google-generative-ai', ...)`. For thinking-level
  models (2.5 Pro/Flash), map the level string to the effort string.
- **Generation config:** map `generationConfig.temperature`, `maxOutputTokens`
  (→ `maxTokens`), etc., into the options fragment.

**Verify:** unit tests — system instruction, user/model turns, `inlineData`
image, `functionCall`, and especially the `functionResponse`-in-user-turn →
`ToolResultMessage` split; tool declarations; thinkingConfig mapping.

### T5.2 — Outbound serialiser `beta/context-to-gemini.ts`

**New file.** Study `transformers/gemini/stream-formatter.ts` and
`stream-transformer.ts` before implementing (`DESIGN.md` → Stage 4 outbound).

- **Non-streaming:** `{ candidates: [{ content: { role:"model", parts:[...] },
  finishReason, ... }], usageMetadata: { promptTokenCount, candidatesTokenCount,
  totalTokenCount } }`. Parts: text → `{ text }`; thinking → `{ thought: true,
  text }` (Gemini 2.5 format); tool calls → `{ functionCall: { name, args } }`.
- **Streaming — NDJSON, not SSE:** emit one Gemini response object per line.
  Each `text_delta` → a line `{ candidates: [{ content: { parts: [{ text:
  delta }] } }] }`. The final line includes `usageMetadata`. Frames are
  newline-delimited JSON; the serialiser yields `JSON.stringify(obj) + "\n"`
  per line (no `data:` prefix, no `[DONE]`).

**Verify:** unit tests — non-streaming shape (text, thinking, tool call);
NDJSON streaming line-by-line including the terminal usage line.

### T5.3 — Beta Gemini route + streaming detection

**File:** `beta/index.ts` (extend `registerBetaRoutes`).

- Register the Gemini-shaped path mirroring `gemini.ts`
  (`/v1beta/models/:modelWithAction`, L28). **Pick the beta URL prefix
  deliberately** — see open decision; the cleanest is
  `/beta/v1beta/models/:modelWithAction`.
- Parse the model name and action from `:modelWithAction`; derive `streaming =
  modelWithAction.includes('streamGenerateContent')` (matches `gemini.ts:97`).
  Pass `streaming` explicitly to the executor (Gemini does **not** have a body
  `stream` flag).
- Standard scaffold: requestId + header, `debug.startLog`, quota middleware,
  abort wiring, parse via `geminiRequestToContext`, call executor with
  `incomingApiType: 'gemini'`.
- **Response content type for streaming is NDJSON** (`application/json` stream /
  `text/plain`-style line stream as the existing Gemini path uses) — not
  `text/event-stream`. Match the existing `gemini.ts` streaming response
  headers.
- Error shape: Gemini error shape (match `gemini.ts`).
- `onSuccess` is a no-op for Stage 4.

**Verify:** `:generateContent` → JSON; `:streamGenerateContent` → NDJSON; 400
when no registry-valid beta-compatible candidate remains.

## Open decisions

- **Beta route prefix for Gemini.** Gemini's native path is already
  `/v1beta/...`, so a literal `/beta` + `/v1beta` reads oddly. Options:
  (a) `/beta/v1beta/models/:modelWithAction` (consistent with the other beta
  routes' `/beta` prefix — recommended); (b) a distinct prefix. Recommended:
  (a) for consistency and so the "configure the upstream URL with `/beta`"
  opt-in story (per `DESIGN.md` → Beta Parallel Path Pattern) works uniformly.
- **URL image / reasoning-detection** — same standing decisions as earlier
  stages.

## Risks / gotchas

- **NDJSON ≠ SSE.** The serialiser and route must emit newline-delimited JSON
  with no `data:` framing and no `[DONE]` sentinel. Reusing the SSE helper here
  is a bug.
- **`functionResponse` inside `user` turns** is the highest-risk mapping. A
  `user` turn can contain a mix of real user parts and `functionResponse`
  parts; the parser must route the latter to `ToolResultMessage` and not bury
  them as user text. Verify against `part-mapper.ts`.
- **Streaming detection is URL-driven**, not body-driven — the executor's
  `streaming` flag must come from the action suffix.
- **Thinking parts** use `{ thought: true }` in Gemini 2.5; don't emit plain
  text for thinking deltas.
- **Schema compatibility** — Gemini's OpenAPI-style tool schemas mostly map
  through `jsonSchemaToTypeBox`, but watch for `type` casing / unsupported
  keywords; add coverage if a real tool schema fails.
- **Executor reuse** — missing capability belongs in the shared executor (M2).
