# M4 — Stage 3: `/beta/v1/responses` (OpenAI Responses API)

**Depends on:** M2 (executor + beta pattern). **Blocks:** M6 Stage-3 production promotion.

The Responses API has a different input/output shape *and* server-side state
(`previous_response_id`, `conversation`, post-response storage). The executor is
still reused unchanged — the new wrinkle is that **state loading happens in the
route handler before parsing**, and **post-response storage happens via the
executor's `onSuccess` hook** (the first non-trivial use of that hook).

Design references: `DESIGN.md` → "Stage 3" and "Responses API state storage in
beta". Existing code to study/borrow: `routes/inference/responses.ts`
(`normalizeInput` ~L75, `previous_response_id`/`conversation` loading ~L89–129,
`body.store` storage ~L221–231), `ResponsesStorageService`
(`services/responses-storage.ts`: `getResponse:165`, `getConversation:260`,
`storeResponse:88`, `updateConversation:207`), and `ResponsesTransformer`
(`formatStream`/`transformStream` for the exact streaming event sequence).

---

## Goal & exit criteria

Done when:

- `beta/responses-to-context.ts` converts the (already-concatenated) Responses
  `input` array → pi-ai `Context` + `ProviderStreamOptions`.
- `beta/context-to-responses.ts` emits the Responses non-streaming object and
  the Responses streaming event sequence.
- `POST /beta/v1/responses` is registered, loads `previous_response_id` /
  `conversation` context **before** parsing (reusing the existing handler's
  logic), calls the M2 executor with `incomingApiType: 'responses'`, and stores
  the response via `onSuccess` when `body.store !== false`.
- Unit + integration tests pass.
- Gates: `bun run typecheck`, `bun run test`, `bun run format:check`.

## Tasks

### T4.1 — Inbound parser `beta/responses-to-context.ts`

**New file.** Converts the Responses `input` array → pi-ai `Context`
(`DESIGN.md` → Stage 3 inbound). By the time this runs, `body.input` already
contains the full concatenated history (the route does state loading first — see
T4.3), so this parser is pure shape-mapping.

- **Input normalization:** borrow `normalizeInput()` logic from `responses.ts`
  (~L75) for the string → array conversion. Items are `type: "message"` with
  `role` + `content[]` whose entries have types `input_text`, `input_image`,
  `output_text`, `tool_call`, `tool_result`.
- **Message mapping:** `role:"user"` + `input_text` → `UserMessage`;
  `role:"assistant"` + `output_text` → assistant history; `role:"assistant"` +
  `tool_call` → assistant with `ToolCallContent`; tool-result items →
  `ToolResultMessage`.
- **System:** no top-level `system`; collect `role:"system"` items into
  `context.systemPrompt`.
- **Tools:** `jsonSchemaToTypeBox` (M1.7).
- **Reasoning:** Responses uses `reasoning: { effort, summary }`; map `effort`
  via `buildReasoningOptions` (M1.5). Track whether `summary` was requested so
  the serialiser can emit reasoning summaries (T4.2).

**Verify:** unit tests — string input, message/tool_call/tool_result mapping,
system collection, reasoning effort pass-through.

### T4.2 — Outbound serialiser `beta/context-to-responses.ts`

**New file.** Study `ResponsesTransformer.formatStream()` and
`transformStream()` for the exact event sequence before implementing
(`DESIGN.md` → Stage 3 outbound).

- **Non-streaming:** `{ id, object: "response", status: "completed", model,
  output: [...], usage: { input_tokens, output_tokens, ... } }`. Output items:
  text block → `{ type:"message", role:"assistant", content:[{ type:
  "output_text", text }] }`; reasoning content → `{ type:"reasoning", summary:[{
  type:"summary_text", text }] }` (only when summary requested); tool calls →
  `{ type:"function_call", call_id, name, arguments }`.
- **Streaming:** the Responses event schema is verbose. Emit the sequence
  `response.created` → `response.output_item.added` →
  `response.content_part.added` → `response.output_text.delta` →
  `response.output_text.done` → `response.content_part.done` →
  `response.output_item.done` → `response.completed`. Match the exact ordering
  and ids used by `ResponsesTransformer.formatStream()`.

**Verify:** unit tests — non-streaming shape (text, reasoning-summary, tool
call) and the full streaming event sequence.

### T4.3 — Route `POST /beta/v1/responses` (with state)

**File:** `beta/index.ts` (extend `registerBetaRoutes`; it will need a
`ResponsesStorageService` instance — construct one like `responses.ts:24`).

Handler order (mirror `responses.ts`, then diverge to the executor):
1. requestId + `x-request-id`; `debug.startLog`; quota middleware; abort wiring.
2. **State loading BEFORE parsing** (`responses.ts` ~L89–129): if
   `previous_response_id`, `getResponse(...)`; if missing → Responses-shaped
   error. If `conversation`, `getConversation(...)`; prepend its items to
   `normalizeInput(body.input)`. After this, `body.input` holds full history.
3. Parse via `responsesToContext(body)` (T4.1).
4. Call the M2 executor with `incomingApiType: 'responses'` and an **`onSuccess`
   hook** that performs post-response storage: when `body.store !== false`
   (note: existing handler also guards `!body.stream` for storage — preserve
   that), call `responsesStorage.storeResponse(formattedResponse, body)` and, if
   `body.conversation`, `responsesStorage.updateConversation(...)`
   (`responses.ts` ~L221–231). The hook receives the `AssistantMessage`; build
   the stored response object from it (reuse T4.2's non-streaming serialiser).
5. Write JSON or Responses SSE.
6. Error shape: OpenAI Responses error shape (as in `responses.ts`).

**Verify:** `previous_response_id` not-found → correct error; a stored
non-streaming response is retrievable via the existing `getResponse` path;
conversation prepend works.

## Open decisions

- **Storage of streaming responses.** The existing handler stores only
  non-streaming (`body.store !== false && !body.stream`, `responses.ts:221`).
  Decide whether the beta path should additionally store streaming responses
  using the reconstructed `AssistantMessage` (the executor already reconstructs
  one for debug, T2.6/T2.7). Recommended: **match existing behaviour** (store
  non-streaming only) to stay zero-regression; revisit later. Owner: `DESIGN.md`
  → "Responses API state storage in beta".
- **`onSuccess` signature** — it receives the `AssistantMessage`. Confirm the
  M2 executor passes enough context (model name, usage) for `storeResponse` to
  build a complete stored object; if not, widen the hook payload in M2 rather
  than reaching into executor internals here.
- **URL image / reasoning-detection** — same standing decisions as Stages 1–2.

## Risks / gotchas

- **State loading must precede parsing.** If you parse `body.input` before
  prepending conversation/previous-response history, the model loses context.
  Order matters (T4.3 step 2 before step 3).
- **`onSuccess` is the only storage path in beta.** Unlike the legacy handler
  (which stores after `handleResponse`), the beta path has no `handleResponse`;
  forgetting to wire `onSuccess` silently drops persistence.
- **Responses streaming is the most verbose schema** of all stages — get the
  event ordering and `output_index`/`content_index` bookkeeping right by
  matching `ResponsesTransformer.formatStream()`.
- **Don't double-load state.** The beta route loads state itself; it must
  bypass `Dispatcher.dispatch()` entirely (executor path), so there's no second
  loader to coordinate with.
- **Executor reuse** — as with M3, missing capability belongs in the shared
  executor, not a Stage-3 fork.
